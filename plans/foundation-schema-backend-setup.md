# Plan: Foundation, Schema, Backend Setup

> Source PRD: `docs/prd/foundation-schema-backend-setup.md`

## Architectural decisions

Durable decisions that apply across all phases:

- **Routes**: This foundation exposes only `GET /health`, returning `{ "status": "ok", "timestamp": "<instant>" }`. No business APIs are introduced in this plan.
- **Schema**: PostgreSQL is the source of truth. Flyway owns migrations. UUID primary keys are generated by PostgreSQL with `gen_random_uuid()`. `created_at` and `updated_at` are database-managed UTC `timestamptz` values.
- **Table naming**: The application user table is physically named `app_users` to avoid confusion with Supabase Auth and database roles.
- **Enums**: Domain enums are stored as text columns with database `CHECK` constraints. Values are explicit: roles `owner/admin/pharmacist/staff`, confidence `low/medium/high`, reorder status `ok/amber/red`, purchase order status `draft/reviewed/sent/cancelled`, CSV upload status `pending/processing/completed/failed`, safety posture `conservative/balanced/aggressive`, notification type `critical_reorder/amber_reorder/daily_digest/weekly_insight/csv_upload_completed/csv_upload_failed/purchase_order_draft`, drug status `active/inactive/unknown`, chat role `user/assistant/system`.
- **Drug identity**: Drug-linked tenant tables store `din text` and enforce foreign keys to `drugs(din)`. DIN values preserve leading zeros and are never cast to integers.
- **Deletes**: Foreign keys use restrictive delete behavior by default. Location lifecycle uses `deactivated_at`.
- **Tenant configuration**: `drug_thresholds` is unique per `(location_id, din)` and `notification_settings` is unique per organization.
- **Forecast history**: Forecast reruns are allowed; multiple rows may exist over time for the same location, DIN, and horizon.
- **Key models**: Organizations, locations, app users, drugs, dispensing records, forecasts, drug thresholds, stock adjustments, purchase orders, notifications, CSV uploads, chat messages, and notification settings.
- **Auth**: No auth flow is implemented in this foundation. RLS SQL assumes `app_users.id` equals the Supabase Auth `auth.uid()` UUID. Spring Boot must still validate organization and location ownership in future services.
- **RLS artifact**: Supabase RLS is provided as a dedicated SQL artifact. Public drug reads are allowed; tenant rows are scoped through `app_users.organization_id`; service-role compatibility is documented without moving authorization responsibility out of Spring Boot.
- **External services**: No Python forecast service, LLM service, Resend, Stripe, or frontend integration is implemented. Spring Boot remains the future orchestration and enforcement layer.
- **Compliance**: `patient_id` is nullable opaque text on dispensing records only. It must not be sent to LLM services, prompts, logs, exports, chat context, or generated purchase orders.
- **Local development**: Local PostgreSQL may be provided through Docker Compose. Dev seed data runs only in the local/dev profile path, never in production.
- **Testing**: Use TDD in small vertical slices. Prefer behavior-level tests through public seams: health endpoint, application startup, Flyway migration validity, and repository/schema compatibility.

## Phase 1: Bootable Backend Skeleton

**User stories**: 1, 2, 3, 4, 26

### TDD seam

Drive this slice through the public HTTP interface: `GET /health`.

First red-green cycle:

- **RED**: write one behavior-level test that calls `GET /health` and expects HTTP 200, `status: "ok"`, and a non-empty timestamp.
- **GREEN**: add only enough Spring Boot application, configuration, security allowance, and controller code to make that test pass.
- **REFACTOR**: only after green, clean package placement and response DTO naming if needed.

### What to build

Create the smallest bootable Spring Boot backend foundation that proves the project can start, expose a health check, and run under local and production-style configuration boundaries. This slice establishes Maven, Java 21, the required Spring dependencies, package boundaries, baseline configuration, and a behavior-level test for the health endpoint.

### Acceptance criteria

- [ ] The project builds with Maven on Java 21.
- [ ] The backend uses Spring Boot 3 and includes the required web, security, validation, actuator, data JPA, OAuth2 resource server, PostgreSQL, Flyway, and Lombok dependencies.
- [ ] The package structure includes the planned domain and infrastructure packages.
- [ ] Local and production profiles exist with clear datasource and Flyway behavior.
- [ ] `GET /health` returns `status: "ok"` and a timestamp.
- [ ] A behavior-level test proves the health endpoint response contract.
- [ ] No business endpoints are introduced.

## Phase 2: Core Tenant Model Migration

**User stories**: 5, 6, 7, 20, 21, 22, 23, 24, 26

### What to build

Add the tenant foundation that every future workflow depends on: organizations, locations, application users, and organization-level notification settings. This slice proves Flyway can create database-managed UUIDs and timestamps, restrictive tenant relationships, baseline enums, indexes, repositories, and the first RLS policy shape.

### Acceptance criteria

- [ ] Flyway creates the PostgreSQL extensions, timestamp trigger function, and core tenant tables.
- [ ] Organizations, locations, app users, and notification settings include UUID primary keys and database-managed UTC timestamps.
- [ ] Locations belong to organizations and support `deactivated_at`.
- [ ] `app_users.id` is compatible with Supabase Auth UUID identity.
- [ ] Role values are constrained with a text `CHECK` constraint.
- [ ] Notification settings are unique per organization.
- [ ] JPA entities and repositories exist for the tenant tables.
- [ ] Indexes support tenant lookup and notification settings access where useful.
- [ ] RLS SQL documents organization and location-scoped access patterns for these tables.
- [ ] A migration test verifies the schema applies cleanly to PostgreSQL.

## Phase 3: Drug and Dispensing History Model

**User stories**: 8, 9, 10, 20, 21, 22, 25, 26

### What to build

Add the drug catalog and dispensing history path needed for future CSV ingestion and demand forecasting. This slice makes DIN the stable catalog identifier, enforces drug references from dispensing records, stores historical quantities and optional patient IDs, and adds local seed data that demonstrates a minimal pharmacy dataset.

### Acceptance criteria

- [ ] Drugs are stored with unique DIN and catalog metadata.
- [ ] Dispensing records belong to a location and reference an existing drug DIN.
- [ ] Dispensing records store dispensed date, dispensed quantity, quantity on hand, optional exact unit cost, and optional patient ID.
- [ ] Patient ID is not represented in any seed output, prompt, export, or future-facing LLM structure in this phase.
- [ ] Indexes support `(location_id, din, dispensed_date)` dispensing history lookups and DIN catalog lookup.
- [ ] JPA entities and repositories exist for drugs and dispensing records.
- [ ] Dev-only seed data creates one organization, one location, one owner user, five sample drugs, and sample dispensing records.
- [ ] Tests verify migrations and repository/schema compatibility for drug and dispensing tables.

## Phase 4: Forecast and Inventory Risk Persistence

**User stories**: 11, 12, 13, 14, 20, 21, 22, 26

### What to build

Add the persistence model for future on-demand forecasts and inventory risk decisions without implementing forecasting itself. This slice records generated forecasts, threshold configuration, and stock adjustments while preserving the architecture rule that Spring Boot persists forecasts and Python services compute only numeric forecast outputs.

### Acceptance criteria

- [ ] Forecasts belong to a location and reference an existing drug DIN.
- [ ] Forecasts store generation time, horizon, predicted quantity, confidence, days of supply, reorder status, Prophet interval values, average daily demand, reorder point, and data points used.
- [ ] Forecast horizon and quantity values are integer-based.
- [ ] Forecast metric values use exact numeric types with appropriate scale.
- [ ] Confidence and reorder status are constrained text enum values.
- [ ] Drug thresholds belong to a location and DIN, include default lead-time and risk settings, and constrain safety posture values.
- [ ] Stock adjustments belong to a location and DIN and store adjustment quantity, adjusted time, and notes.
- [ ] Indexes support forecast and stock adjustment timeline queries by location and DIN.
- [ ] JPA entities and repositories exist for forecasts, drug thresholds, and stock adjustments.
- [ ] Tests verify migration validity and JPA schema compatibility for forecast/inventory tables.

## Phase 5: Operational Workflow Persistence

**User stories**: 15, 16, 17, 18, 19, 20, 21, 22, 26

### What to build

Add the persistence model for future operational workflows: purchase order drafting, notifications, CSV upload tracking, and location-scoped chat history. This slice creates the storage contracts only; it does not send notifications, parse CSVs, call Grok, or generate purchase orders.

### Acceptance criteria

- [ ] Purchase orders belong to a location and store generated time, Grok output text, optional JSONB line items, and constrained status.
- [ ] Notifications belong to an organization, may optionally belong to a location, and store constrained type, JSONB payload, sent time, and read time.
- [ ] CSV uploads belong to a location and store filename, constrained status, optional error message, row count, drug count, validation summary, and upload time.
- [ ] Chat messages belong to a location and store role and content.
- [ ] JSONB fields are used for structured payloads, validation summaries, and purchase order line items.
- [ ] Indexes support CSV upload timelines, notification timelines, and chat timelines.
- [ ] JPA entities and repositories exist for purchase orders, notifications, CSV uploads, and chat messages.
- [ ] Tests verify migration validity and JPA schema compatibility for operational workflow tables.

## Phase 6: Foundation Hardening Pass

**User stories**: 23, 24, 26, 27

### What to build

Tighten the foundation so later feature work can proceed safely. This slice reviews RLS SQL coverage, verifies profile behavior, confirms no out-of-scope endpoints or integrations slipped in, and ensures tests make the core contracts executable.

### Acceptance criteria

- [ ] Supabase RLS SQL covers public drug reads and tenant-owned table access for organization and location-scoped rows.
- [ ] RLS SQL documents the backend service-role-compatible model while preserving the rule that Spring Boot still enforces authorization.
- [ ] Production Flyway configuration excludes dev seed migrations.
- [ ] Local/dev configuration includes seed migrations.
- [ ] Tests cover health endpoint behavior, application startup, Flyway migration validity, and schema compatibility.
- [ ] The final codebase contains no auth flow implementation, CSV parsing, forecast integration, LLM integration, notification delivery, billing integration, or business APIs.
- [ ] The foundation can be handed off to the next feature with clear domain boundaries and no hidden generated patient data exposure.
